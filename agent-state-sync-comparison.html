tmux-scout 与 Flux Desktop 的 agent 状态同步：同一类事件，两种产品边界

这份对比基于当前本地代码：`/Users/bytedance/repos/tmux-scout` 与 `/Users/bytedance/repos/flux-desktop-app`。核心结论是：`tmux-scout` 是 tmux 内的轻量 agent 状态索引器，优先解决「看见、排序、跳转、发现崩溃」；`Flux Desktop` 是桌面端状态中枢， 把同样的 hook 信号扩展成审批、问答、通知、远程、统计和 UI 生命周期。

TL;DR tmux-scout 把状态压缩成 fzf/status-bar 里的可信索引；Flux 把状态提升成可交互的产品状态机。
tmux hook → optional bridge → JSON files → picker/status；subagent 多数作为父会话摘要，用户仍回终端处理。
Flux hook/pull → BridgeServer → AgentEvent → SessionState → IPC；可阻塞 hook 并回写 approval/question directive。
impact tmux-scout 更少打扰、更少依赖；Flux 更强可视化与远程/通知能力，但一致性边界和竞态处理明显更复杂。

一句话结论

两边现在已经在「hook 优先、reducer 归一、JSON/transcript 兜底、隐藏内部子会话」这些方向上趋同，但产品目标不同导致取舍不同。

定位差异

tmux-scout 是导航层

它不试图接管 agent 交互，只把多 agent、多 pane、多状态变成可扫视的索引。用户看见 `WAIT` / `BUSY` / `DONE` 后跳回 tmux pane。

状态差异

Flux 是产品状态层

它把原始 hook 变成 `AgentEvent`，再变成 `AgentSession`。这个 session 直接驱动 Island、Pet、审批卡、问答卡、通知和统计。

Subagent 差异

同样隐藏子会话，颗粒度不同

两者都避免 subagent 污染顶层列表。Flux 对 Claude/Codex/Coco/Hermes 的父子关系和工具活动更完整；tmux-scout 目前 Codex/Kimi 较完整，Claude 只记录事件。

维度 tmux-scout Flux Desktop App 对用户的影响
产品边界 tmux 插件，面向 fzf picker 和 status bar。状态是「索引」和「跳转线索」。 Electron 桌面应用，面向 Island / Pet / Settings。状态是驱动 UI 和外部行为的中心事实。 tmux-scout 不会打断终端工作流；Flux 能主动提醒和处理，但用户会感受到更多 UI 状态变化。
入口协议 各 agent hook 脚本直接写 `~/.tmux-scout/status.json` 和 `sessions/*.json`，watchdog 可启动 `bridge.sock` 串行化写入。 所有 hook 通过 `BridgeServer` 的 Unix socket 进入，协议有 `hello`、`processHook`、`resolvePermission`、`answerQuestion`。 tmux-scout 即使后台 bridge 不在也能继续记录；Flux 依赖主应用 socket，但能对 hook 返回可执行 directive。
状态模型 统一 reducer 有 `idle`、`running`、`waitingForApproval`、`waitingForAnswer`、`completed`、`interrupted`、`crashed`、`stale`，并保留 legacy `status`。 `AgentSession.phase` 更少，只有 `running`、`waitingForApproval`、`waitingForAnswer`、`completed`，其余靠 `isSessionEnded`、`isProcessAlive`、`completionDismissed`、`isRemote` 等 flag 表达。 tmux-scout 的状态标签更适合终端列表；Flux 的状态更适合组合 UI 卡片，但 flag 之间的一致性更难维护。
Subagent 顶层可见性 Codex 子会话会被标记 `isHiddenFromScout`，不进入 picker 顶层；父会话写 `activeSubagents` 摘要。Kimi 也维护父会话子列表。 子会话写 `parentSessionId`，`isVisibleInIsland()` 对子会话返回 false；父会话渲染 `SubagentList`。 两边都减少「子 agent 刷屏」。Flux 给用户可展开列表和耗时；tmux-scout 给用户一行摘要，更省空间。
Claude Subagent 安装 `SubagentStart` / `SubagentStop` hook，但当前 Claude hook 仅写 `lastEvent`，不维护 `activeSubagents` 明细。 ClaudeAdapter 过滤子进程 hook，缓存 `Agent` / `Task` 描述，`SubagentStart` 写 `SubagentInfo`，子进程工具活动回写父会话。 Claude 多 Task 场景下，Flux 用户能看到子任务列表和最近工具；tmux-scout 用户主要看到父会话状态，子任务细节仍靠回 pane。
Codex Subagent 读取 transcript 首行 `session_meta` 或 hook payload 元信息，按 `forked_from_id` / `thread_spawn.parent_thread_id` 判定子会话；隐藏子会话并汇总到父会话。 SessionStart 同步/异步读取 `session_meta`，`UserPromptSubmit` 再兜底；建立 `parentSessionMap`，子会话 Stop 时向父会话发 `subagentStopped`。 两者都能避开 Codex 内部线程噪音。Flux 对异步 meta 晚到、Stop 竞态和父会话 UI 清理做得更细，用户更少看到闪烁占位。
审批与问答 只记录 `needsAttention` 和 `pendingToolUse`，picker 显示 `W:APP` / `W:ANS` 等。用户需要跳到终端处理。 BridgeServer 维护 `pendingPermissions` / `pendingQuestions`，可从 Island 或 Feishu 回写；也支持 terminal-native 模式保留卡片。 tmux-scout 更忠实、更少误操作面；Flux 能显著减少切回终端，但 pending 清理、断线策略和审批模式会影响信任感。
崩溃与中断 watchdog/picker sync 做 tmux pane、PID、process tree、Codex JSONL、Claude transcript 的校正；高置信源有优先级保护。 AppCoordinator 每 2 秒做 Claude ESC、Codex transcript、Codex idle 兜底，并结合 PidWatcher / RemotePidWatcher。 tmux-scout 更适合回答「哪个 pane 卡住了」；Flux 更适合让 UI 计时停表、隐藏过期卡片、对远程断连给出反馈。
持久化策略 状态文件长期作为主数据源，`status.json` 聚合，`sessions/*.json` 分片，24 小时清理。 主状态在内存 Map；`saveSessions` 最多写 50 条、24 小时内本地 session，云端会话不落盘；启动时主动清空旧保存。 tmux-scout 重启后更容易保留最近上下文；Flux 更少陈旧历史和隐私残留，但 app 重启后需要 hook/poll 重新填充 UI。
远程场景 主要锚定 tmux pane 与本机文件。hook 会采集终端上下文，但没有完整远端隧道/远程 PID 状态模型。 通过 hook payload 的主机/IP/SSH client 识别远程，管理远程 host 状态、远端 PID 轮询、远程跳转和断连事件。 在纯 tmux/SSH 内，tmux-scout 低成本；在 IDE Remote-SSH、桌面通知、远端进程探活场景，Flux 用户得到更完整的跨机器体验。
UI 输出 一行列表和 status-right 计数，信息密度高：状态、agent、window、project、title、tool/subagent 详情。 渲染 `AgentSession[]`，支持 SessionRow、ApprovalCard、QuestionCard、CompletionCard、SubagentList、usage row。 tmux-scout 适合键盘快速扫视；Flux 适合把状态变成可读、可点、可通知的「任务面板」。

Subagent 深入对比

Subagent 是两边最能体现产品边界差异的地方：同样是「别让子会话污染顶层」，tmux-scout 选择压缩，Flux 选择建模。

tmux-scout

把子会话折叠为父会话的一行状态

  • Codex 子会话：`classifyCodexSession()` 识别 `forked_from_id` 与 `thread_spawn`，`markHiddenSession()` 标记隐藏，并 `upsertParentSubagent()`。
  • picker 渲染时用 `subagentDetail()` 取最新一个子 agent，展示类似 `1 subagent · reviewer: Bash: npm test`。
  • Kimi 的 `SubagentStart` / `SubagentStop` 通过 generic hook 维护 `activeSubagents`。
  • Claude 虽安装 subagent hooks，但当前 handler 只记录 `subagent_start` / `subagent_stop` 事件，不维护子列表。
Flux Desktop

把子 Agent 作为父 session 的一等展示对象

  • `SubagentInfo` 是共享类型，包含 `agentId`、`agentType`、`taskDescription`、`transcriptPath`、`startedAt`、`lastToolActivity`。
  • ClaudeAdapter 会过滤子进程普通 hook，只把工具活动写回父会话，避免子进程独立上岛。
  • CodexAdapter 建立 `parentSessionMap`，子 session 自己有 `parentSessionId`，但 Island 不独立展示。
  • UI 层 `SubagentList` 显示最多 3 条、支持展开，并实时显示 elapsed 与最近工具活动。

Subagent 对用户的实际差别

这不是简单的信息多少问题，而是「用户是否需要理解多 agent 拓扑」。

噪音控制

两边都做了隐藏

Codex 内部 title generation、ambient suggestions、memory writing、guardian reviewer 等后台会话不会作为独立任务打扰用户。

进度解释

Flux 更能解释「为什么还在跑」

当父 agent 长时间没有主输出，但子 agent 在跑搜索或验证，Flux 的 `SubagentList` 能解释等待原因；tmux-scout 目前只给短摘要。

交互边界

tmux-scout 更不接管

子 agent 的审批/问题在 tmux-scout 里仍回终端处理。Flux 对主会话审批更强，但对 Claude 子进程权限会选择静默 ACK，不把所有子进程交互都搬到 Island。

用户影响汇总

把工程差异翻译成用户语言，可以归纳成四个体验轴。

1. 可预期性

`tmux-scout` 更像被动仪表盘：它告诉你哪里需要看，但不改变 agent 行为。对重度 tmux 用户，这种可预期性很高。

tmux-scout
Flux

2. 可操作性

`Flux` 的优势是把「待审批」「待回答」「完成」「远程断开」直接变成卡片、按钮、声音和 Feishu 通知。

tmux-scout
Flux

3. 多 agent 可解释性

Flux 对 subagent、cloud agent、remote agent 建模更多，用户更容易理解系统在并行做什么。tmux-scout 适合把这些压成可扫视的短行。

tmux-scout
Flux

4. 故障表面积

tmux-scout 的依赖链短，坏了通常只是状态不刷新；Flux 的链路更长，坏了可能表现为卡片不消失、审批失效、通知误报或远程状态不准。

tmux-scout
Flux
设计判断： 如果要让 `tmux-scout` 继续向 Flux 对齐，最值得借鉴的是「更完整的 Claude subagent 汇总」和「typed event contract」； 最不应该照搬的是「从 UI 直接接管审批/问答 directive」。后者会改变 tmux-scout 的产品边界，让它从导航插件变成交互代理。

面向 tmux-scout 的改进建议

如果目标是提升状态同步质量，同时保持 tmux 插件的轻量边界，建议按收益和复杂度分层。

高收益

补齐 Claude subagent 摘要

复用 Flux 的思路：在 `PreToolUse(Agent/Task)` 缓存 description，`SubagentStart` 写 `activeSubagents`，子进程工具活动回写父会话。UI 仍只显示一行摘要。

中收益

收敛通用 AgentEvent 命名

当前 tmux-scout 已有 `lastEvent` 与 `lifecycleEvent`，可以逐步把 hook adapter 的输出收敛为更显式的内部事件对象，减少各 hook 自己拼状态字段。

谨慎

不要急着加 UI 审批

一旦要从 picker 回写 approval/question，就需要 pending map、socket 生命周期、directive schema、断线策略和安全边界。这已经是 Flux 的产品复杂度。

可复制摘要

用于同步给团队时，可以直接复制下面的短版结论。

短版： `tmux-scout` 和 `Flux Desktop` 都在走 hook-first 状态同步，并都需要处理 Codex/Claude 这类 agent 的内部线程、subagent、审批等待和中断兜底。 但 `tmux-scout` 的边界是 tmux 导航索引，所以状态最终被压缩成 fzf/status-bar 的短行，用户仍回终端处理； `Flux` 的边界是桌面产品状态中枢，所以状态会进入 typed `AgentEvent`/`AgentSession`、IPC、UI 卡片、通知、远程探活和 directive 回写。 对用户来说，tmux-scout 更轻、更可预期；Flux 更可操作、更能解释复杂多 agent/远程状态，但也承担更多竞态和 stale pending 风险。
